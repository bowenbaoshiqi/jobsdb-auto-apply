# JobsDB 简历投递助手 v2.0 设计文档

- **日期**: 2026-07-20
- **状态**: 待审阅
- **作者**: brainstorming 协作产出
- **范围**: v2.0 = 纯重构 + TDD + 环境修复,不加新功能

## 1. 目标与背景

### 1.1 目标

v1.0 功能已上线可正常投递简历。v2.0 在**不新增业务功能、不改投递行为**的前提下,达成三件事:

1. **按 TDD 模式开发** — 建立快速、不依赖浏览器的单元测试体系
2. **增强健壮性** — 消除异常吞没,让失败可见、可重试、可降级
3. **降低耦合性** — 抽象浏览器层,解耦 Orchestrator,拆分 God Object

### 1.2 v1.0 现状诊断

- **规模**: ~5000 行源码,16 个模块,15 个测试文件
- **环境脆弱**: `.venv/bin/python3` 是断掉的 symlink(指向已删除的 miniconda base),系统 python3 无依赖,**当前没有任何可用 Python 环境能跑测试**
- **测试金字塔倒置**: 15 个测试里 7 个是 e2e(需真实网络+手动登录),纯单元仅 8 个
- **无 CI**: 没有 GitHub Actions、没有 pre-commit
- **God Object**: `Orchestrator.__init__` 硬 new 了 10 个依赖;`apply_flow.py` 543 行(最大文件)
- **异常吞没**: 7 处 `except Exception: pass`(apply_flow 3 处、login 3 处、job_detail 1 处)
- **测试依赖浏览器**: `conftest.py` 顶层 `browser_engine` fixture 强制起 Chromium,任何测试都被迫依赖真实浏览器

### 1.3 决策汇总(brainstorming 阶段已定)

| # | 决策点 | 选择 |
|---|---|---|
| 1 | 推进方式 | 绞杀者模式(v1.0 保持可用,逐步重构) |
| 2 | 第一刀边界 | 浏览器抽象层(`BrowserPort` + `PageController`) |
| 3 | 环境 + CI | uv 重建 .venv + GitHub Actions 跑纯单测 |
| 4 | e2e 测试 | 隔离到 `tests/e2e/` + `@pytest.mark.e2e`,CI 不跑 |
| 5 | v2.0 范围 | 只重构不加功能,投递行为与 v1.0 一致 |
| 6 | 抽象粒度 | 中等:`BrowserPort`(生命周期) + `PageController`(Page 包装) |
| 7 | 重构安全网 | 先特征化测试锁定行为,再重构 |

| 10 | 节奏方案 | 方案 A:自底向上(环境→安全网→抽象→核心→DI→收尾) |

## 3. 浏览器抽象层(BrowserPort + PageController)

### 3.1 BrowserPort — 浏览器生命周期

```python
# src/browser/ports/browser_port.py
from typing import Optional, Protocol

class BrowserPort(Protocol):
    """浏览器抽象:只管生命周期,不管页面操作。"""

    async def start(self) -> "PageController":
        """启动浏览器,返回主页面的 PageController。"""
        ...

    async def stop(self) -> None:
        """停止浏览器,保存 cookies/profile。"""
        ...

    async def new_page(self) -> "PageController":
        """新建标签页,返回其 PageController(自动注入 stealth)。"""
        ...

    @property
    def current_page(self) -> Optional["PageController"]:
        """当前活跃页面(未启动返回 None)。"""
        ...
```

只有 4 个方法:v1.0 `BrowserEngine` 的公开方法收敛。`goto` 移到 `PageController`(导航是页面操作)。Cookie/profile 持久化是 `PlaywrightBrowser` 内部细节,不上接口。

### 3.2 PageController — 页面操作(核心)

`jobsdb/*` 和 `simulation/*` 唯一依赖的接口。基于 v1.0 对 `Page` 的全部调用收敛:

```python
# src/browser/ports/page_controller.py
from typing import Optional, Any, Protocol
from playwright.async_api import ElementHandle  # 类型标注用,运行时不耦合

class PageController(Protocol):
    """页面操作抽象:jobsdb/* 和 simulation/* 只依赖这个接口。"""

    # --- 导航 ---
    async def goto(self, url: str, wait_until: str = "domcontentloaded") -> None: ...
    @property
    def url(self) -> str: ...

    # --- 查询(只读) ---
    async def query_selector(self, selector: str) -> Optional[ElementHandle]: ...
    async def query_selector_all(self, selector: str) -> list: ...
    async def is_visible(self, selector: str) -> bool: ...
    async def count(self, selector: str) -> int: ...
    async def get_text(self, selector: str) -> Optional[str]: ...
    async def get_attribute(self, selector: str, attr: str) -> Optional[str]: ...

    # --- 交互(写) ---
    async def click(self, selector: str, timeout: float = 30.0) -> None: ...
    async def fill(self, selector: str, value: str) -> None: ...
    async def type_text(self, selector: str, value: str, delay: int = 50) -> None: ...
    async def select_option(self, selector: str, value: str) -> None: ...
    async def check(self, selector: str) -> None: ...

    # --- 等待 ---
    async def wait_for_selector(self, selector: str, timeout: float = 30.0) -> None: ...
    async def wait_for_timeout(self, ms: int) -> None: ...

    # --- JS 执行 ---
    async def evaluate(self, expression: str) -> Any: ...

    # --- 截图(异常诊断用) ---
    async def screenshot(self, path: str) -> None: ...
    def is_closed(self) -> bool: ...
```

### 3.3 设计要点

1. **接口形式**: `typing.Protocol`(结构子类型)。实现类不需要显式继承,`PlaywrightPageController`/`FakePageController` 只要方法签名对上即可。最 Pythonic,与现有 `BrowserEngine`(未继承抽象基类)兼容。

2. **ElementHandle 处理**: `query_selector` 直接返回 Playwright `ElementHandle`(类型标注)。**类型层耦合 Playwright,但运行时不耦合** —— 假实现可返回 `unittest.mock.AsyncMock` 造的 `ElementHandle`,Protocol 只检查结构。实用权衡:类型提示清晰,测试用 mock 不必自建 Element 抽象体系。

3. **便利方法**: `get_text`/`get_attribute`/`count`/`is_visible` 收敛 v1.0 里到处出现的 `elem = await page.query_selector(s); text = await elem.inner_text() if elem else None` 三行模式。假实现只需返回预设值。

4. **`type_text` 而非 `type`**: 避开 Python 内建 `type` 名字冲突。`delay` 参数保留,`simulation/typing.py` 拟人打字仍可在外层包装调用。

5. **不暴露 `page.on()` 事件**: v1.0 没用到,YAGNI。

### 3.4 两个实现

- **`PlaywrightPageController`**: 包装真实 `Page`,每个方法委托给 Playwright。`BrowserEngine` 降级为其内部细节(或被 `PlaywrightBrowser` 取代)。
- **`FakePageController`**: 纯 Python 内存对象,持有 `dict[selector, element]`。测试预设"这个 selector 存在/可见/文本是什么",`jobsdb/*` 逻辑不碰浏览器地跑。

### 3.5 收益对照

v1.0 测 `apply_flow` 必须起 Chromium。v2.0 后:

```python
# tests/unit/test_apply_flow.py
async def test_apply_returns_captcha_when_captcha_detected():
    fake_page = FakePageController()
    fake_page.set_visible(SELECTORS.RECAPTCHA_IFRAME, True)
    flow = ApplyFlow(fake_page)
    result = await flow.apply("job-123")
    assert result.status == ApplyStatus.CAPTCHA
    # 不起任何浏览器,毫秒级
```

## 4. apply_flow.py 拆分(543 行 God Object → 状态机 + 步骤处理器)

### 4.1 现状

`apply_flow.py` 543 行,含 `ApplyStep` 状态机枚举 + `apply()` 主循环 + 大量 `_detect_*`/`_handle_*` 私有方法 + 3 处 `except Exception: pass`。

### 4.2 拆分策略

保持状态机骨架,把每个 step 的处理逻辑拆成独立类:

```
src/jobsdb/apply/
├── __init__.py
├── flow.py            # ApplyFlow 状态机骨架(apply 主循环,~120 行)
├── step_base.py       # StepHandler Protocol: detect(page)->bool / handle(page)->ApplyStep
├── steps/
│   ├── __init__.py
│   ├── captcha_check.py     # _check_captcha 独立
│   ├── popup_dismiss.py     # _dismiss_popups 独立
│   ├── resume_step.py       # RESUME_SELECTION 处理
│   ├── questions_step.py    # QUESTIONS 处理
│   ├── cover_letter_step.py # COVER_LETTER 处理(选 Don't include)
│   ├── review_step.py       # REVIEW 处理
│   └── submit_step.py       # SUBMITTED 检测 + 成功确认
└── detectors.py      # _detect_current_step / _check_success 等纯查询逻辑
```

### 4.3 StepHandler 协议

```python
# src/jobsdb/apply/step_base.py
from typing import Protocol
from src.browser.ports.page_controller import PageController
from src.jobsdb.apply.flow import ApplyStep

class StepHandler(Protocol):
    """单个申请步骤的处理器。"""
    async def detect(self, page: PageController) -> bool:
        """当前页面是否处于此步骤。"""
        ...

    async def handle(self, page: PageController, human) -> ApplyStep:
        """处理此步骤,返回下一个预期步骤。"""
        ...
```

### 4.4 ApplyFlow 骨架瘦身后

```python
# src/jobsdb/apply/flow.py
class ApplyFlow:
    def __init__(self, page: PageController, human=None, max_steps=10,
                 handlers: list[StepHandler] = None):  # 可注入
        self.page = page
        self.human = human
        self.max_steps = max_steps
        self.handlers = handlers or default_handler_chain()  # 按优先级

    async def apply(self, job_id: str) -> ApplyResult:
        # 主循环:遍历 handlers,第一个 detect()==True 的负责 handle
        # ~120 行,只含循环 + 状态推进 + 超时/计数
        ...
```

### 4.5 收益

| 项 | v1.0 | v2.0 |
|---|---|---|
| 单文件行数 | 543 | 流骨架 ~120 + 每个 step ~40-60 |
| 可测性 | 必须起浏览器 | 每个 StepHandler 用 FakePageController 独立测 |
| `except Exception: pass` | 3 处 | 0 |
| 加新步骤 | 改 543 行大文件 | 加一个 `steps/xxx_step.py` + 注册 |

### 4.6 行为不变保证

拆分是纯机械重构:`_handle_cover_letter` 逻辑原样搬到 `CoverLetterStep.handle`,selectors 不动,HumanSimulator 调用不动。特征化测试(第 6 节)在拆分前后对比,确保 `apply()` 对相同输入返回相同 `ApplyResult`。

## 5. Orchestrator 工厂模式 DI

### 5.1 问题

v1.0 `Orchestrator.__init__` 硬 new 10 个依赖(Database/ApplyQueue/RateLimiter/TimingOptimizer/ApplicationTracker/AlertManager/StatsAggregator + browser/login/scraper 延迟创建)。无法注入假实现 → 难测试。

### 5.2 工厂抽象

```python
# src/factory.py
from typing import Optional, Protocol

class ComponentFactory(Protocol):
    """生产 Orchestrator 所需组件的工厂。10 个独立方法,一一对应 v1.0 依赖。"""
    def create_browser(self, account_alias: str) -> BrowserPort: ...
    def create_database(self, db_path: str) -> Database: ...
    def create_queue_manager(self, db, config): ...
    def create_rate_limiter(self, config, db): ...
    def create_timing_optimizer(self, config): ...
    def create_tracker(self, db): ...
    def create_alert_manager(self, config): ...
    def create_stats(self, db): ...
    def create_login_handler(self, page, ...): ...
    def create_scraper(self, page, ...): ...

class DefaultFactory:
    """生产真实组件 — 等价于 v1.0 行为。"""
    def create_browser(self, account_alias):
        return PlaywrightBrowser(...)
    def create_database(self, db_path):
        return Database(db_path)
    # ... 其余直接 new 真实类

class FakeFactory:
    """生产内存假组件 — Orchestrator 单测用。"""
    def create_browser(self, account_alias):
        return FakeBrowser()
    def create_database(self, db_path):
        return InMemoryDatabase()
    # ...
```

工厂放 `src/factory.py`(顶层,因为它跨 browser/db/scheduler/monitor,不只 browser)。10 个独立方法不分组(一一对应,清晰胜于简短)。

### 5.3 Orchestrator 改造

```python
class Orchestrator:
    def __init__(self, config=None, account=None, max_jobs=None,
                 factory: Optional[ComponentFactory] = None):  # 新增
        self.config = config or get_config()
        self.account = account or Account(alias="default", email="", password="")
        self.max_jobs = max_jobs or ...
        self.factory = factory or DefaultFactory()  # 默认真实,可注入假

        # 组件走工厂(不再硬 new)
        self.db = self.factory.create_database(self.config.storage.database_path)
        self.db.set_account(self.account.alias)
        self.queue_manager = self.factory.create_queue_manager(self.db, self.config.scheduler)
        self.rate_limiter = self.factory.create_rate_limiter(self.config.scheduler, self.db)
        # ... 其余 8 个同理

        self.browser: Optional[BrowserPort] = None  # 延迟创建,沿用 v1.0 时机

    async def _init_browser(self):
        self.browser = self.factory.create_browser(self.account.alias)
        self.page = await self.browser.start()  # 返回 PageController
        self.human = HumanSimulator(self.page)
        self.login_handler = self.factory.create_login_handler(self.page, ...)
        self.scraper = self.factory.create_scraper(self.page, ...)
```

### 5.4 收益

| 项 | v1.0 | v2.0 |
|---|---|---|
| `Orchestrator()` 测试 | 必须起浏览器+真实 DB | `Orchestrator(factory=FakeFactory())` |
| `run()` 主流程测试 | 不可行 | 注入假工厂,验证协调逻辑 |
| 调用方改动 | — | `main.py`/`scripts` 仍 `new Orchestrator()`,零改动 |
| 依赖方向 | 硬依赖 10 个具体类 | 依赖 1 个 ComponentFactory 接口 |

### 5.5 行为不变保证

`DefaultFactory` 每个方法生产 v1.0 原本 `__init__` 里 new 的同一个类、同样参数。`run()` 的 6 阶段流程(`_init_browser` → `_ensure_login` → `_scrape_jobs` → `build_queue` → `_process_queue` → `_create_session_report`)完全不变。

## 6. 特征化测试 + 测试基础设施(安全网)

### 6.1 特征化测试

特征化测试锁住 v1.0 当前行为,重构后仍绿 = 行为未变。测纯逻辑、高价值、不碰浏览器的模块:

| 模块 | 锁定内容 |
|---|---|
| `selectors.py` | 选择器常量值(快照) |
| `storage/database.py` | CRUD 行为(存取、去重、状态流转) |
| `scheduler/queue.py` | `build_queue` 过滤/排序 |
| `scheduler/queue.py` | `RateLimiter` 频率计算 |
| `simulation/mouse.py` | Bezier 路径属性(连续、非直线、有界) |
| `simulation/typing.py` | 延迟分布属性 |
| `accounts/registry.py` | add/list/use/show 行为 |
| `jobsdb/apply_flow.py` | ApplyStep 枚举 + 状态转移(用 mock page) |

**原则**: 特征化测试先于重构写。对 `apply_flow`:重构前用 mock Page 写特征化测试锁定"给这个 mock 页面 → apply() 返回这个 ApplyResult";拆分后跑同一批测试,绿 = 没改行为;重构后这些测试升级为正式单元测试。

## 7. 异常处理 + 健壮性收尾

### 7.1 问题清单

7 处 `except Exception: pass`:

### 7.2 处理策略:三分法

每处按语义归入三类:

**A. 该重试 → `@retry` 装饰 + 捕获具体异常**
```python
@retry(max_attempts=3, base_delay=2.0, exceptions=(ElementNotFoundError, TimeoutError))
async def _click_popup_dismiss(self, page):
    await page.click(MODAL_CLOSE)
```

**B. 该降级(非致命)→ 捕获具体异常 + `logger.warning`/`debug`**
```python
try:
    salary = await page.get_text(JOB_DETAIL_SALARY)
except ElementNotFoundError:
    salary = None
    logger.debug(f"Salary not found for job {job_id}")
```

**C. 该上抛(致命)→ 移除 try/except,让异常按体系传播**
```python
# _check_success 失败不能静默,移除 except 让 ApplyError/CaptchaDetectedError 上抛
```

### 7.3 边界异常约定

`jobsdb/*` 模块只抛 `JobsDBError` 子类(不抛裸 Exception/RuntimeError)。`Orchestrator.run()` 顶层按体系捕获:

```python
try:
    ...  # 6 阶段流程
except CaptchaDetectedError as e:
    return self._create_error_report("captcha", str(e))  # 触发通知,等人工
except SessionExpiredError:
    return self._create_error_report("session_expired", ...)  # 清 cookies
except RateLimitError:
    return self._create_error_report("rate_limited", ...)  # 长等待
except JobsDBError as e:
    return self._create_error_report("error", str(e))
# 不捕获 Exception —— 未预期异常带完整 traceback 退出,不静默
```

这是 v1.0 `Orchestrator.run()` 已有模式的细化(把笼统 `except Exception` 改成按 JobsDBError 子类分流)。
