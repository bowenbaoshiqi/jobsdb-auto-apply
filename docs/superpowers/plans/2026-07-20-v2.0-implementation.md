# v2.0 实施计划

- **日期**: 2026-07-20
- **对应 spec**: `docs/superpowers/specs/2026-07-20-v2.0-design.md`
- **节奏**: 方案 A(自底向上),6 阶段,每阶段独立验收
- **原则**: 绞杀者模式 — v1.0 保持可用,每步可回滚

## 通用约定

- **分支**: 所有工作在 `feature/v2.0` 分支进行(从 `feature/v1.0` 切出),保护 v1.0
- **TDD 循环**: 每个代码改动遵循 红(写失败测试)→ 绿(最小实现)→ 重构
- **提交粒度**: 每个可独立验收的步骤一次提交,提交信息带阶段前缀如 `phase0:`
- **回滚**: 每阶段若失败,`git reset --hard <阶段起始 commit>` 即可回到 v1.0 可用状态
- **安全网**: 阶段 1 的特征化测试在阶段 2-5 始终保持绿 = 行为未变

## 阶段间依赖与并行性

```
阶段0 (环境) ──> 阶段1 (安全网) ──> 阶段2 (抽象) ──> 阶段3 (核心重构)
                                                          │
                                                          v
                                              阶段4 (DI) ──> 阶段5 (收尾)
```

- 阶段 0-3 严格串行(后者依赖前者)
- 阶段 4 依赖阶段 2(需要 BrowserPort 抽象)和阶段 3(jobsdb 改造完)
- 阶段 5 依赖阶段 3、4(异常处理需在重构后的代码上做)
- 任何阶段停下,v1.0(`feature/v1.0` 分支)仍完整可用

## 提交规范

- 前缀: `phase0:` / `phase1:` / ... / `phase5:`
- 例: `phase0: 修复 .venv (uv) + 隔离 e2e 测试`
- 每个可独立验收的步骤一次提交
- 阶段结束打 tag: `git tag v2.0-phase0` ... `v2.0-phase5`

## 风险与回滚

- **最高风险**: 阶段 3(动核心投递逻辑)
- **安全网**: 阶段 1 的特征化测试 + 阶段间的 e2e 手动验证
- **回滚**: 任何阶段失败,`git reset --hard v2.0-phase<N-1>` 回到上一阶段
- **终极回滚**: `git checkout feature/v1.0` 回到完全可用的 v1.0
